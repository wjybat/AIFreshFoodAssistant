## 1. 背景与动机

### 1.1 传统超市的"清库存"困局

生鲜品类是超市的流量引擎，也是损耗重灾区。行业平均生鲜损耗率在 5%~10%，部分叶菜高达 15% 以上。面对临期、滞销，超市的传统手段几乎只有一招：

> **打折。** 越临期越打折，最后 5 折、3 折，甚至直接报废。

这条路径的问题不是"没用"，而是它**把商品仍然当成商品在卖**——顾客看到的依然是"一堆快过期的青椒"，而不是"今晚的一顿饭"。打折只能吸引价格敏感型顾客，且严重侵蚀毛利、损害品牌价格心智。

### 1.2 消费者的"今晚吃什么"决策真空

几乎每个家庭每天都面临同一个问题：**今晚吃什么？** 这不是价格问题，而是**决策成本问题**。消费者缺的不是商品，缺的是一个"理由"——一个把零散食材组织成"一顿饭"的理由。

### 1.3 机会洞察

把上面两个问题拼在一起，就是本产品的核心洞察：

> **超市的库存问题，本质是"组织方式"问题。**
> 卖不动的不是青椒，而是"没有被组织进一道菜里的青椒"。
> 如果 AI 能每天把当天库存重新组织成"今晚想吃的菜"，就能同时解决供给端损耗和需求端决策。

这就是从**库存逻辑**到**生活场景逻辑**的范式转换。

### 1.4 与同类方案的差异化

| 维度 | 传统生鲜损耗预测 | AI 优惠券/补货 | **本产品：餐桌预测引擎** |
|------|------------------|----------------|--------------------------|
| 核心逻辑 | 预测哪些会损耗 → 提前打折 | 预测需求 → 发券/补货 | LLM 把库存重组成消费场景 |
| 技术路径 | 统计模型 + 规则 | 推荐算法 + 规则 | **LLM Agentic Workflow（生成式推荐）** |
| 输出形态 | 损耗报表 + 打折建议 | 优惠券 / 补货单 | **菜谱 + 场景包 + 全渠道触达 + 扫码菜谱页** |
| 顾客视角 | "便宜的快过期菜" | "一张券" | "今晚 15 分钟做青椒肉丝" |
| 解决痛点 | 单一（损耗） | 单一（销量） | **六合一**（见 §3.3） |

## 2. 产品定位

### 2.1 定位陈述

**面向社区型超市的"即时烹饪场景经营大脑"**：每天综合商品库存与临期状况（优先考量）、客流、天气、日期节日、下班放学时间等社区信息，由 LLM Agentic Workflow 自动生成"今晚吃什么"场景包——包括最佳商品品类搭配组合候选及对应菜谱候选，驱动备货、陈列、熟食加工、会员触达与门店运营全链路。

### 2.2 核心决策逻辑

本产品**不做显式的组合优化或约束求解**，而是采用**生成式推荐**范式：

- LLM 接收当日全量上下文（库存/临期/天气/社区/节日/通勤等）；
- 结合保存的 **memory（历史成功样例）** 和 **skills（领域知识沉淀）**；
- 通过 **few-shot 提示词**引导 LLM 调用其自身世界知识（菜谱常识、食材搭配、口味场景、营养均衡等）；
- **推理生成**最佳商品品类搭配组合候选及其对应能组成的菜谱候选。

这不是"从固定菜谱库中检索匹配"，而是让 LLM 像一个有经验的厨师长一样，**看着今天的食材和天气，即兴创造今晚的菜单**。

### 2.3 目标用户（B 端）

| 角色 | 关注点 |
|------|--------|
| 超市店长 / 运营 | 降损耗、提客单、省人力 |
| 生鲜采购 / 供应链 | 备货精准、周转健康 |
| 会员运营 / 市场 | 复购、到店、App 活跃 |
| 熟食 / 预制菜部门 | 产能匹配、销量提升 |
| 总部商品/品类 | 连带率、品类协同、数据沉淀 |

### 2.4 最终消费者（C 端）

社区家庭顾客——收到"今晚 15 分钟做青椒肉丝"的推送，买到搭配好的备菜包，扫码即可查看完整菜谱步骤，省去"想吃什么"的纠结。

## 3. 核心价值与痛点解决

### 3.1 范式转换

```
传统：卖不动的菜  →  打折  →  低价清掉（损毛利、伤心智）
本产品：卖不动的菜  →  LLM 重组为"今晚的菜"  →  创造消费理由（保毛利、提连带）
```

### 3.2 价值闭环

```
库存/临期 + 社区信息 ──► LLM Agentic Workflow ──► 场景化套餐 + 菜谱
         (天气/客流/节日/通勤)        (生成式推荐)          │
                                                            │
              ┌─────────────────────────────────────────────┤
              ▼                       ▼                     ▼
        App 推送（引流）        门店陈列（转化）       熟食联动（连带）
              │                       │                     │
              │                  扫码看菜谱                  │
              │                  (云端菜谱页)                │
              │                       │                     │
              └───────────► 顾客到店购买场景包 ◄────────────┘
                                      │
                              数据回流 → Memory 沉淀 → 优化下一轮
```

### 3.3 六合一痛点解决

| # | 痛点 | 本产品如何解决 |
|---|------|----------------|
| 1 | 生鲜损耗 | 临期商品优先进入当日菜谱，以"场景"而非"打折"形态售出 |
| 2 | 连带销售 | LLM 生成的套餐天然组合多品类（主菜+配菜+调料+主食+汤） |
| 3 | 会员复购 | 每日个性化推送"今晚菜单"，养成打开习惯 |
| 4 | 熟食/预制菜销售 | LLM 推理联动熟食区（+N 元得米饭/汤），带动熟食区 |
| 5 | 门店陈列 | LLM 输出陈列建议与拣货指令，员工即装即用 |
| 6 | "今晚吃什么" | 替消费者做决策，扫码即可查看完整菜谱，降低选择成本 |

### 4.2 数据接入层（输入）

| 数据源 | 字段示例 | 用途 | 优先级 |
|--------|----------|------|:---:|
| 临期预警 | 商品、剩余保质期 | **最高优先**：临期商品必须优先进入菜谱候选 | ★★★ |
| 库存系统 | 商品、库存量、周转天数 | **高优先**：高库存/滞销商品优先消耗 | ★★★ |
| 生鲜 IoT | 温湿度、品相评分 | 判断食材是否还能进入菜谱 | ★★ |
| 天气 API | 天气、温度、体感 | 影响菜品偏好（雨天热汤、高温凉拌） | ★★ |
| 客流感知 | 时段客流预测 | 备货量与推送时机 | ★★ |
| 社区画像 | 半径3km客群构成、家庭/白领比 | 菜品口味与份量 | ★★ |
| 节假日日历 | 节日、节气 | 节令菜品（冬至饺子、端午粽） | ★★ |
| 通勤节奏 | 学校放学、写字楼下班时间 | 推送与备菜时机 | ★ |
| 历史销售 | 热销菜 TOP、连带品类 | 作为 few-shot 样例输入 LLM | ★★ |

> **决策优先级**：商品临期状况 > 库存水位 > 天气/社区/节日等场景因素。LLM 在推理时以此优先级为软约束，自然地在生成结果中体现。

### 4.3 AI 决策层（核心：LLM Agentic Workflow）

这是本产品与传统方案最大的差异点。**不使用显式的组合优化或约束求解算法**，而是通过 LLM + Harness 框架实现生成式推荐。

#### 4.3.1 Harness 框架（编排层）

Harness 框架负责 Agentic Workflow 的编排调度，不替代 LLM 的推理能力，而是为 LLM 提供：

- **上下文组装**：将当日库存、临期、天气、社区等多源数据组装为结构化 prompt 上下文；
- **工具调用编排**：LLM 可按需调用工具（库存查询、天气API、菜谱生成、菜谱页面部署等）；
- **多步推理调度**：支持 LLM 分步思考（先分析临期 → 再匹配场景 → 再生成组合 → 再生成菜谱）；
- **输出解析**：将 LLM 的自然语言输出解析为结构化业务对象（菜单、套餐、触达内容等）；
- **Memory 读写**：每次生成后，将成功样例写入 Memory；每次生成前，检索相关历史样例作为 few-shot。

#### 4.3.2 Memory（历史成功样例沉淀）

Memory 是 LLM 的"经验记忆"，存储历史生成中**被验证成功的样例**：

- **样例结构**：`{日期, 输入上下文(库存/天气/社区), 生成结果(菜单/套餐), 实际效果(售出率/客单价/连带率), 是否成功}`
- **检索方式**：按当前场景与历史样例的相似度（天气相似、库存结构相似、社区画像相似）检索 Top-K 样例；
- **用途**：作为 few-shot 示例注入 prompt，引导 LLM 参考过去成功经验生成当前方案；
- **自学习**：每日数据回流后，自动标注成功/失败样例，持续丰富 Memory。

#### 4.3.3 Skills（领域知识沉淀）

Skills 是 LLM 的"专业能力包"，以结构化提示词形式沉淀零售与烹饪领域知识：

- **菜谱知识 Skill**：食材搭配常识、口味平衡原则、烹饪时间与难度评估、份量换算；
- **场景适配 Skill**：天气→菜品偏好的映射经验（雨天热菜、高温凉拌、节日传统菜）、客群→份量/口味的适配规则；
- **零售经营 Skill**：毛利保护原则（套餐价不低于成本线）、品类连带逻辑（主菜+配菜+汤+主食的均衡）、陈列与拣货的业务约束；
- **触达文案 Skill**：App推送文案风格、门店屏展示要点、员工指令的清晰度要求。

LLM 在推理时按需激活相关 Skill，相当于一个有经验的厨师长+店长在同时思考。

#### 4.3.4 Few-Shot 提示词

每次生成时，Harness 从 Memory 中检索 Top-K 历史成功样例，构造 few-shot prompt：

```
[System]
你是一个社区超市的"AI 厨师长+经营大脑"。你的任务是根据今日库存/临期状况（最高优先）、
天气、社区画像、节日、通勤节奏等信息，生成今日推荐菜单及商品搭配组合。

决策原则：
1. 临期商品必须优先进入菜谱候选
2. 高库存商品优先消耗
3. 套餐需品类均衡（主菜+配菜+汤/主食）
4. 菜品需适配今日天气与社区客群
5. 套餐定价需保护毛利（不低于成本线）

[Skills: 菜谱知识 / 场景适配 / 零售经营 / 触达文案]
（此处注入相关 Skill 的结构化知识）

[Few-Shot: 历史成功样例 ×K]
样例1: {输入: 雨天/青椒临期/猪肉高库存/家庭客群, 输出: 青椒肉丝套餐+麻婆豆腐+...}
样例2: {输入: 高温/黄瓜临期/冬瓜高库存/白领客群, 输出: 凉拌黄瓜木耳+冬瓜排骨汤+...}
...

[今日输入]
库存: 青椒50kg(1天临期), 猪肉80kg(偏高), 豆腐30kg(2天临期), ...
天气: 下雨18°C
社区: 家庭客群, 18:00下班高峰
节日: 无
历史热销: 青椒肉丝, 麻婆豆腐

[输出要求]
请生成3-4套推荐菜单候选，每套包含：菜品名、所需食材及分量、套餐定价、适配理由、
触达文案、员工指令。同时为每道菜生成完整菜谱（步骤+贴士），供扫码查看。
```

#### 4.3.5 LLM 推理生成（生成式推荐）

LLM 接收上述 prompt 后，利用自身世界知识进行推理生成：

- **食材→菜谱的创造性映射**：不是从固定库检索，而是 LLM 基于烹饪常识推理"青椒+猪肉丝能做什么菜"，生成菜谱候选；
- **场景适配推理**：LLM 理解"雨天→热菜需求↑""家庭客群→2人份"等因果关系，自然在生成中体现；
- **组合均衡性判断**：LLM 自主判断生成的多道菜是否品类均衡、口味互补；
- **文案与指令生成**：同一轮推理中，LLM 一并生成推送文案、门店屏内容、员工指令、菜谱步骤。

**这是"生成式推荐"而非"检索式推荐"**——LLM 不是从预设菜谱库中选，而是根据今日食材即兴创造，因此能覆盖菜谱库中不存在的搭配组合，具备更强的泛化能力和创新性。

## 7. 技术实现方案

### 7.1 技术范式：LLM + Harness Agentic Workflow

本产品的核心技术路径是**纯 LLM 驱动的生成式推荐**，不依赖传统的组合优化（如 OR-Tools）或约束求解器。

| 组件 | 选型 | 说明 |
|----|------|------|
| LLM | 大语言模型（如 GPT-4 / Claude 等） | 核心推理引擎，负责菜谱生成、组合搭配、文案生成 |
| Harness 框架 | Agent 编排框架（如 LangChain / 自研 harness） | 管理 Agent 生命周期、工具调用、上下文、输出解析 |
| Memory | 向量数据库 + 结构化存储 | 存储历史成功样例，支持相似度检索 |
| Skills | 结构化提示词模板 | 领域知识包，按场景激活 |
| Few-Shot | 动态构造 | 每次从 Memory 检索 Top-K 样例注入 prompt |
| 工具集 | API 封装 | 库存查询、天气查询、菜谱页面部署、二维码生成等 |
| 前端 | Web 应用（云端部署） | 运营驾驶舱 + 扫码菜谱页 |
| 后端 | API 服务（云端部署） | 编排 Harness、管理 Memory、部署菜谱页面 |

### 7.2 为什么选择生成式推荐而非组合优化

| 维度 | 传统组合优化（OR-Tools等） | **LLM 生成式推荐（本产品）** |
|------|--------------------------|------------------------------|
| 菜谱来源 | 固定菜谱库中检索 | LLM 即兴创造，覆盖库中不存在的搭配 |
| 泛化能力 | 受限于菜谱库规模 | 利用 LLM 世界知识，无限菜谱空间 |
| 场景理解 | 需人工编码规则（雨天→热菜） | LLM 自然语言理解，零规则编码 |
| 文案生成 | 需额外模型/模板 | 同一轮推理中生成 |
| 可解释性 | 数学优化目标 | LLM 推理链（自然语言，人类可读） |
| 迭代成本 | 改规则需改代码 | 改 prompt 即可，灵活迭代 |
| 自学习 | 需重训模型 | Memory 持续积累，few-shot 自动优化 |

### 7.4 关键实现细节

#### 7.4.1 Memory 检索

```python
# 伪代码：Memory 检索
def retrieve_memory(current_context, top_k=5):
    # current_context: {weather, inventory_summary, community, festival, ...}
    # 在向量数据库中检索相似历史样例
    query_embedding = embed(current_context)
    similar_cases = vector_db.search(query_embedding, top_k=top_k)
    # 过滤：只保留标注为"成功"的样例
    successful_cases = [c for c in similar_cases if c.is_successful]
    return successful_cases
```

#### 7.4.2 Prompt 组装

```python
# 伪代码：Harness 组装 prompt
def build_prompt(today_data, memory_cases, skills):
    system = f"""
    你是社区超市的AI厨师长+经营大脑。
    决策优先级：临期商品 > 高库存 > 天气/社区/节日适配。
    {skills['recipe_knowledge']}
    {skills['scene_adaptation']}
    {skills['retail_operation']}
    """
    few_shot = "\n".join([format_case(c) for c in memory_cases])
    user = f"""
    今日输入：{format_input(today_data)}
    请生成3-4套推荐菜单候选...
    """
    return system, few_shot, user
```

### 8.2 场景 S1 详解

**输入上下文**：
- 天气：今天下雨，18°C，体感偏凉
- 库存：青椒剩余 50kg（1 天临期）、猪肉库存偏高 80kg、豆腐 30kg（2 天临期）、米饭类熟食今日销量下降 40%
- 社区：周边 3km 以家庭客群为主，写字楼下班高峰 18:00
- 客流：预计今日 850 人
- 历史热销：青椒肉丝、麻婆豆腐

**LLM 推理链**（Harness 可视化）：
1. 分析临期：青椒 1 天临期 + 猪肉库存偏高 → 推理可组成"青椒肉丝"（世界知识：青椒+猪肉丝是经典搭配）
2. 分析临期：豆腐 2 天临期 → 推理可组成"麻婆豆腐"（世界知识：豆腐+猪肉末+豆瓣酱）
3. 场景适配：雨天体感凉 → 热菜需求↑；米饭熟食销量↓ → 联动熟食区出米饭/汤品
4. 客群适配：家庭客群 + 18:00 下班 → 推荐 2 人份备菜包，17:30 推送
5. 组合均衡：青椒肉丝(主菜)+麻婆豆腐(配菜)+紫菜蛋花汤+熟食米饭 → 品类均衡

**输出**：
- 推荐菜单：青椒肉丝（主）+ 麻婆豆腐（配）+ 紫菜蛋花汤 + 熟食米饭
- 每道菜完整菜谱（食材分量+步骤+贴士）→ 部署为云端菜谱页 → 生成二维码
- 套餐：青椒肉丝 2 人份备菜包，原价¥32 → 套餐价¥25.8
- 联动：+¥6 得熟食米饭+例汤
- App 推送："今晚 15 分钟做青椒肉丝 🌧️ 雨天就该吃口热乎的"（含菜谱扫码入口）
- 门店屏：菜谱大图 + 制作步骤 + 套餐价 + **菜谱二维码**
- 员工指令：青椒/猪肉丝拣货 → 场景堆头布置 → 熟食区备饭
- 价值预估：青椒预计 100% 售出（减损¥240）、客单价+18%、熟食区客流+35%

### 9.2 商业化路径

| 阶段 | 模式 | 说明 |
|------|------|------|
| 试点 | SaaS 订阅 + 门店数计费 | 单店月费，含引擎+触达+驾驶舱+扫码菜谱 |

### 9.3 竞争壁垒

1. **Memory 数据飞轮**：历史成功样例越多，few-shot 质量越高，生成结果越好——数据积累形成正向循环；
2. **生成式推荐能力**：不依赖固定菜谱库，LLM 即兴创造的能力远超检索式系统；
3. **全链路闭环**：从数据→LLM推理→多端触达→扫码菜谱→数据回流，不是单点工具；
4. **场景叙事**：差异化定位"把库存变成一顿饭"，认知壁垒强。

| 凉拌黄瓜木耳 | 黄瓜、木耳 | 蒜、醋、香油 | 高温天、开胃 |
| 手工水饺 | 面粉、猪肉馅、白菜 | 韭菜、姜 | 节日、家庭 |

> 注：以上仅为示例。实际运行时，LLM 可生成不限于上述菜品的任意菜谱组合——这是生成式推荐与检索式推荐的核心区别。

## 附录 B：术语表

| 术语 | 含义 |
|------|------|
| Agentic Workflow | LLM 作为自主智能体，在 Harness 框架编排下多步推理完成复杂任务 |
| Harness 框架 | Agent 编排层，管理上下文组装、工具调用、输出解析、Memory 读写 |
| 生成式推荐 | LLM 基于世界知识即兴生成推荐结果，而非从固定库中检索 |
| Memory | 历史成功样例的存储，作为 few-shot 检索源 |
| Skills | 领域知识包（菜谱/场景/零售/文案），以结构化提示词形式注入 |
| Few-Shot | 从 Memory 检索的历史成功样例，作为 prompt 示例引导 LLM |
| 场景包 | 围绕"一顿饭"组织的商品组合+定价+触达内容+扫码菜谱 |
| 扫码菜谱页 | 部署在云端的独立菜谱网页，二维码指向其 URL，任意设备可访问 |
